# 项目宪章 Constitution

## 1. 技术栈

| 层级 | 技术选择 |
|------|----------|
| 前端框架 | Vue 3 (Composition API) |
| 后端框架 | Express.js |
| API 风格 | REST + GraphQL (双协议支持) |
| 数据库 | SQLite (MVP 阶段) |
| 运行方式 | 直接 Node.js 运行 |

## 2. 代码规范

### 2.1 语言规范
- **后端**: TypeScript (strict 模式)
- **前端**: TypeScript (标准模式)
- **命名**: camelCase (变量/函数), PascalCase (组件/类)

### 2.2 代码质量
- ESLint + Prettier (使用 Vue/TypeScript 基础配置)
- 提交前必须通过 ESLint 检查

### 2.3 目录结构

```
web-agent/
├── client/           # Vue 前端
│   ├── src/
│   │   ├── components/
│   │   ├── views/
│   │   ├── api/
│   │   └── stores/
│   └── package.json
├── server/           # Express 后端
│   ├── src/
│   │   ├── routes/
│   │   ├── graphql/
│   │   ├── services/
│   │   └── models/
│   └── package.json
└── package.json      # 根目录 workspace
```

## 3. 开发流程 (TDD)

### 3.1 流程要求
1. 先写测试，再写实现代码
2. 测试覆盖：核心业务逻辑 + API 端点
3. 所有测试必须通过才能提交

### 3.2 测试框架
- 后端: Jest (或 Vitest)
- 前端: Vitest + Vue Test Utils
- E2E: Playwright (可选)

### 3.3 代码提交
- 必须包含对应测试
- 使用conventional commits格式

## 4. 文档规范

- 统一使用 Markdown 格式
- 文档目录: `docs/`
- 核心文档:
  - `SPEC.md` - 需求规格
  - `PLAN.md` - 技术方案
  - `TESTS.md` - 测试用例

## 5. MVP 范围

### 5.1 核心功能
- Web 聊天界面
- 支持 OpenAI 兼容 API 的大模型对接
- 会话历史记录
- 基本的 Agent 指令配置

### 5.2 非 MVP 范围 (后期迭代)
- 用户认证/权限
- 插件系统
- 高级 Agent 编排
- 生产级部署配置
