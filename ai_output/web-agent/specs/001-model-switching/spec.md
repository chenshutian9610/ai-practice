# Feature Specification: 模型切换功能

**Feature Branch**: `001-model-switching`
**Created**: 2026-04-22
**Status**: Draft
**Input**: 用户描述: "聊天框增加一个模型切换的功能：1. 模型列表只在第一次实时查询，然后缓存在当前浏览器，然后在模型搜索框旁边加一个刷新按钮，点击后再重新拉取模型列表 2. 原先在设置那里的模型，定义修改为默认模型，新的聊天窗口默认选中的模型"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 模型列表缓存与刷新 (Priority: P1)

用户点击模型选择器时，系统优先使用浏览器缓存的模型列表。只有在首次使用或用户手动刷新时才从 API 拉取最新列表。

**Why this priority**: 这是核心交互模式变更，用户体验依赖于缓存策略的正确实现。

**Independent Test**: 可以通过点击模型选择器，验证下拉列表是否使用缓存显示，同时测试刷新功能是否重新拉取数据。

**Acceptance Scenarios**:

1. **Given** 用户首次使用模型选择器，**When** 点击模型选择器，**Then** 系统实时查询可用模型列表并缓存到浏览器
2. **Given** 用户已经加载过模型列表，**When** 再次点击模型选择器，**Then** 系统使用缓存的模型列表，无网络请求
3. **Given** 用户查看模型列表，**When** 点击刷新按钮，**Then** 系统重新从 API 拉取模型列表并更新缓存

---

### User Story 2 - 搜索过滤模型列表 (Priority: P1)

用户可以在模型列表中通过搜索框输入关键词，系统实时过滤显示匹配的模型。

**Why this priority**: 当模型数量较多时，搜索功能可以快速定位目标模型，提升用户体验。

**Independent Test**: 可以在模型选择器打开时输入搜索词，验证列表是否正确过滤。

**Acceptance Scenarios**:

1. **Given** 模型列表已显示，**When** 用户在搜索框输入 "gpt"，**Then** 只显示名称包含 "gpt" 的模型
2. **Given** 搜索结果为空，**When** 用户查看列表，**Then** 显示 "未找到匹配的模型" 提示
3. **Given** 用户清空搜索框，**When** 列表刷新，**Then** 显示完整模型列表

---

### User Story 3 - 切换当前会话使用的模型 (Priority: P1)

用户可以选择当前会话使用的模型，选中的模型会用于后续的消息发送。

**Why this priority**: 这是模型切换的最终目的，用户需要在不同模型间切换以完成不同任务。

**Independent Test**: 选择模型后发送消息，验证是否使用了选定的模型。

**Acceptance Scenarios**:

1. **Given** 用户点击模型选择器并看到模型列表，**When** 用户点击选择 "gpt-4"，**Then** 选中的模型显示在选择器中，且后续消息使用该模型
2. **Given** 用户已选择一个模型，**When** 重新打开模型选择器，**Then** 已选中的模型应高亮或标记为当前选择
3. **Given** 用户切换模型后发送消息，**When** AI 回复，**Then** 确认使用了新选择的模型

---

### User Story 4 - 新会话使用默认模型 (Priority: P2)

创建新会话时，自动使用设置中定义的默认模型。

**Why this priority**: 提供开箱即用的体验，减少每次新建会话都需要手动选择模型的重复操作。

**Independent Test**: 在设置中配置默认模型后，创建新会话，验证新会话是否使用了该默认模型。

**Acceptance Scenarios**:

1. **Given** 用户在设置中配置了默认模型 "claude-3"，**When** 用户点击新建会话，**Then** 新会话的模型选择器显示 "claude-3"
2. **Given** 用户没有设置默认模型，**When** 创建新会话，**Then** 使用系统预定义模型（作为默认值）

---

### User Story 5 - 单独会话模型覆盖默认设置 (Priority: P2)

用户可以为单独会话选择不同于默认设置的模型。

**Why this priority**: 允许用户在特定会话中灵活切换模型，实现更精细的控制。

**Independent Test**: 修改单个会话的模型后发送消息，验证该会话使用新选择的模型而非默认模型。

**Acceptance Scenarios**:

1. **Given** 默认模型是 "gpt-3.5"，**When** 用户在会话中切换到 "gpt-4"，**Then** 该会话使用 "gpt-4"，不影响默认模型设置
2. **Given** 用户切换了会话模型后关闭应用，**When** 重新打开该会话，**Then** 模型保持为用户上次选择的那个

---

### Edge Cases

- 首次使用且 API 不可用：显示友好提示 "无法获取模型列表，请检查 API 配置"，并显示重试按钮
- 刷新时 API 请求失败：保留原有缓存数据，显示错误提示 "刷新失败，使用缓存数据"
- 刷新时 API 响应为空：保留原有缓存，显示警告 "未获取到新模型"
- 用户清空缓存后首次加载：正常触发首次加载流程
- 模型列表为空：显示友好提示 "暂无可用模型，请检查 API 配置"
- 用户快速多次点击刷新按钮：防止重复发起请求，使用防抖或禁用状态
- 刷新过程中用户关闭选择器：后台继续请求，更新缓存供下次使用

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在首次点击模型选择器时从 API 拉取模型列表
- **FR-002**: 系统 MUST 将模型列表缓存到浏览器本地存储
- **FR-003**: 系统 MUST 在非首次使用时从缓存加载模型列表
- **FR-004**: 系统 MUST 在模型选择器搜索框旁提供刷新按钮
- **FR-005**: 系统 MUST 在用户点击刷新按钮时重新从 API 拉取模型列表并更新缓存
- **FR-006**: 系统 MUST 支持在模型列表中通过搜索框过滤模型
- **FR-007**: 系统 MUST 在用户选择模型后立即应用到当前会话
- **FR-008**: 系统 MUST 在用户设置中保存默认模型配置
- **FR-009**: 系统 MUST 在创建新会话时使用默认模型作为初始值
- **FR-010**: 系统 MUST 将每个会话的模型选择持久化存储
- **FR-011**: 系统 MUST 在模型列表加载时显示加载状态
- **FR-012**: 系统 MUST 在模型查询失败时显示错误信息并提供重试选项

### Key Entities

- **Model (模型)**: 代表一个可用的 LLM 模型，包含唯一标识符（如 "gpt-4"）和显示名称
- **Session (会话)**: 每个会话记录当前选中的模型 ID
- **Settings (设置)**: 包含默认模型 ID 配置
- **ModelCache (模型缓存)**: 存储在浏览器本地的模型列表数据，包含缓存时间戳

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 首次加载模型列表时，用户可以在 3 秒内看到列表（假设 API 响应正常）
- **SC-002**: 缓存命中时，模型列表应立即显示，无网络延迟
- **SC-003**: 搜索过滤的响应时间小于 100ms（前端子集过滤）
- **SC-004**: 新建会话时默认模型正确填充的成功率达到 100%
- **SC-005**: 用户能够在模型列表中快速找到目标模型（通过搜索或滚动）
- **SC-006**: 模型切换后发送的消息正确使用新模型
- **SC-007**: 刷新操作能够在 5 秒内完成并更新显示

## Assumptions

- 后端 API 支持 `/v1/models` 或类似的模型列表查询端点
- 用户已经在设置中配置了有效的 API Endpoint 和 API Key
- 浏览器支持 localStorage 或 sessionStorage 用于缓存
- 用户的模型数量在合理范围内（不超过几百个），不需要分页处理
- 缓存应包含足够的信息以便在刷新失败时恢复使用
